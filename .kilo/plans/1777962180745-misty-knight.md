# Plan: RaiFlow Startup Modes, API Key Simplification, and SDK Philosophy

## Summary

Implement the foundational architectural decisions that make RaiFlow a true "app-private database-like" runtime:
1. **Startup mode enforcement** — `custodial` or `non-custodial`, required at boot
2. **API key simplification** — `RAIFLOW_API_KEY` is the only required key; remove premature complexity
3. **SDK philosophy enforcement** — PoW is never the developer's concern; low-level escape hatches carry warnings
4. **Documentation alignment** — README, progress, and RFC updates to codify these decisions

---

## Part 1: Startup Mode (`RAIFLOW_MODE`)

### 1.1 Add `mode` to config

**File:** `packages/config/src/index.ts`

- Add `'custodial' | 'non-custodial'` to `DaemonConfig`
- Parse `daemon.mode` from YAML (optional — falls back to env var)
- Resolution order: `RAIFLOW_MODE` env var > `daemon.mode` in YAML > **FAIL with clear error**
- The error message should explain both modes and how to set one

```ts
export interface DaemonConfig {
  host: string;
  port: number;
  apiKey?: string;
  mode: 'custodial' | 'non-custodial';
}
```

### 1.2 Enforce mode at startup

**File:** `packages/runtime/src/main.ts`

- After loading config, validate that `mode` is set. If not, print a clear help message and `process.exit(1)`:
  ```
  [raiflow] RAIFLOW_MODE is required. Set it to "custodial" or "non-custodial".
    custodial:      RaiFlow manages keys, derives accounts, signs blocks, generates PoW.
                    Requires RAIFLOW_CUSTODY_SEED and RAIFLOW_CUSTODY_REP.
    non-custodial:  RaiFlow acts as a relay and monitor. All signing happens client-side.
                    Some features (invoices, managed accounts, sends) are unavailable.
  ```
- Log the mode prominently at startup: `[raiflow] starting in CUSTODIAL mode` or `[raiflow] starting in NON-CUSTODIAL mode`

### 1.3 Gate features by mode

**File:** `packages/runtime/src/handler.ts`

- In `non-custodial` mode, routes that require custody should return `501 Not Implemented` with a clear message:
  - `POST /accounts` (type=managed) → blocked
  - `POST /accounts/:id/sends` → blocked
  - `POST /invoices` → blocked (requires derivation)
  - `POST /work` → still available (relay to node)
  - `POST /blocks` → still available (publish pre-signed)
  - Watched accounts → still available
  - `GET` endpoints → still available

**File:** `packages/runtime/src/runtime.ts`

- Add `mode` to `RuntimeConfig` and store it on the `Runtime` instance
- Guard `createManagedAccount`, `queueSend`, `createInvoice` with mode checks

### 1.4 Custodial mode validation

**File:** `packages/runtime/src/main.ts`

- In `custodial` mode, validate that `config.custody` is present (seed + representative). If missing, fail with:
  ```
  [raiflow] CUSTODIAL mode requires custody.seed and custody.representative.
  Set RAIFLOW_CUSTODY_SEED and RAIFLOW_CUSTORY_REP, or add them to raiflow.yml.
  ```

### 1.5 Update config examples

**Files:**
- `raiflow.yml.example` — add `mode:` field with comment
- `packages/runtime/docker/raiflow.yml` — add `mode: "env:RAIFLOW_MODE"`
- `docker-compose.yml` — add `RAIFLOW_MODE` env var with comment

## Part 2: API Key Simplification

### 2.1 Make `RAIFLOW_API_KEY` the single source

**File:** `packages/runtime/src/auth.ts`

- Simplify `resolveApiKey`:
  1. Check `RAIFLOW_API_KEY` env var → use it
  2. Check `config.daemon.apiKey` (from YAML `env:` ref) → use it
  3. **FAIL** with clear message: `RAIFLOW_API_KEY is required. Set it as an environment variable or in raiflow.yml under daemon.apiKey.`
- Remove auto-generation and `.api-key` file logic entirely
- Remove `getApiKeyFilePath` export

**Rationale:** The developer picks their own key. No surprises, no hidden files.

### 2.2 Update auth.ts tests

**File:** `packages/runtime/src/__tests__/auth.test.ts`

- Update tests to reflect new behavior: no auto-generation, failure when missing

### 2.3 Remove show-api-key utility

**File:** `packages/runtime/src/show-api-key.ts`

- Remove this file (no longer needed when there's no auto-generation)
- Remove from Dockerfile CMD alternatives if referenced

### 2.4 Update Docker & compose

**Files:**
- `Dockerfile` — remove `show-api-key` reference if any
- `docker-compose.yml` — uncomment `RAIFLOW_API_KEY` and mark as required
- `README.md` — update "Required Environment Variables" table: `RAIFLOW_API_KEY` becomes **Yes**

### 2.5 Update SDK client

**File:** `packages/raiflow-sdk/src/client.ts`

- Make `apiKey` required (not optional) in `RaiFlowClientOptions`:
  ```ts
  export interface RaiFlowClientOptions {
    baseUrl: string;
    apiKey: string;  // required
  }
  ```

## Part 3: SDK Philosophy — PoW is Not the Developer's Problem

### 3.1 Add philosophy docstring to WorkResource

**File:** `packages/raiflow-sdk/src/resources/Work.ts`

Add JSDoc to the class and `generate` method:
```ts
/**
 * Low-level work generation.
 *
 * DEVELOPER NOTE: If you find yourself using WorkResource directly,
 * it indicates a missing feature in the RaiFlow SDK. RaiFlow is designed
 * so that PoW generation is completely invisible to the developer.
 * Please open an issue describing your use case.
 */
```

### 3.2 Add philosophy docstring to BlocksResource

**File:** `packages/raiflow-sdk/src/resources/Blocks.ts`

```ts
/**
 * Low-level block publishing for pre-signed blocks.
 *
 * DEVELOPER NOTE: This resource exists for advanced non-custodial flows
 * where blocks are signed client-side (e.g., in a browser wallet).
 * For custodial flows, use SendsResource — RaiFlow handles signing and
 * PoW automatically.
 */
```

### 3.3 Update SDK index.ts exports

**File:** `packages/raiflow-sdk/src/index.ts`

- Add JSDoc to `WorkResource` and `BlocksResource` re-exports explaining the escape-hatch nature

### 3.4 Update README.md SDK section

Add a "Developer Experience Philosophy" section to README.md that explains:
- RaiFlow absorbs PoW, signing, frontier management
- If a developer needs raw RPC or PoW, it's a missing feature
- `blocks.publish()` exists for non-custodial pre-signed flows only

## Part 5: Raiflow.yml Example Overhaul

**File:** `raiflow.yml.example`

Update to reflect the new minimal-easy-start philosophy:

```yaml
# RaiFlow Configuration
#
# REQUIRED: Set RAIFLOW_MODE and RAIFLOW_API_KEY before starting.
#   RAIFLOW_MODE=custodial or RAIFLOW_MODE=non-custodial
#   RAIFLOW_API_KEY=your-secret-key
#
# This file is intentionally minimal. All values below have sane defaults.

daemon:
  host: "0.0.0.0"
  port: 3100
  # mode: "env:RAIFLOW_MODE"        # required: "custodial" or "non-custodial"
  # apiKey: "env:RAIFLOW_API_KEY"    # required

nano:
  rpc: ["env:NANO_RPC_URL"]          # required: at least one Nano node RPC
  ws: []
  work: []

invoices:
  defaultExpirySeconds: 3600
  autoSweep: false

storage:
  driver: "sqlite"
  path: "./raiflow.db"

webhooks: []

logging:
  level: "info"
  format: "pretty"

# Required for custodial mode:
# custody:
#   seed: "env:RAIFLOW_CUSTODY_SEED"
#   representative: "env:RAIFLOW_CUSTODY_REP"
```
