# Plan: Seed Watcher CLI Example

## Overview

A CLI tool under `examples/seed-watcher/` that reads seed configurations from a YAML file, derives Nano accounts, checks for pending funds, and publishes receive blocks — all **through the RaiFlow runtime API**. In sweep mode, it also sends all funds to a configured destination address.

The example demonstrates the **non-custodial pattern**: seeds stay local in the CLI config, RaiFlow runs without custody config, and the CLI uses RaiFlow for account tracking, receivable discovery, work generation, and block publishing. Only signing happens locally.

## Required Runtime & SDK Additions

The current RaiFlow runtime and SDK lack three endpoints that this example needs. The plan includes implementing these.

### 1. `GET /accounts/:id/receivable` — List pending receivable blocks

Returns live pending blocks for an account by querying the Nano RPC. Needed to discover what to receive.

**Runtime handler addition** (`handler.ts`):
```ts
// GET /accounts/:id/receivable
if (method === 'GET' && parts.length === 3 && parts[2] === 'receivable') {
  const account = await runtime.getAccount(accountId);
  if (!account) return errorResponse(`Account not found: ${accountId}`, 'not_found', 404);
  const client = runtime.rpcPool?.getClient();
  if (!client) return errorResponse('RPC not configured', 'bad_request', 400);
  const receivable = await client.accountsReceivable(account.address);
  return json({ data: receivable });
}
```

**SDK addition** (`Accounts.ts`):
```ts
async receivable(id: string): Promise<{ data: Receivable[] }> {
  return this.client.request<{ data: Receivable[] }>('GET', `/accounts/${id}/receivable`);
}
```

**Model addition** (`model/src/index.ts`):
```ts
export interface Receivable {
  hash: string;
  amount: string;
  sender: string;
}
```

### 2. `POST /blocks` — Publish a pre-signed block

Accepts a fully signed block JSON (including work) and publishes it via RPC. This is the air-gapped signing pattern.

**Runtime handler addition** (`handler.ts`):
```ts
// POST /blocks
if (method === 'POST' && parts.length === 1 && parts[0] === 'blocks') {
  const body = await req.json() as Record<string, unknown>;
  const { block } = body;
  if (typeof block !== 'string') {
    return errorResponse('Missing required field: block (JSON string)', 'bad_request', 400);
  }
  const client = runtime.rpcPool?.getClient();
  if (!client) return errorResponse('RPC not configured', 'bad_request', 400);
  const result = await client.process(block);
  return json(result, 201);
}
```

**SDK addition** — new `BlocksResource` class:
```ts
export class BlocksResource {
  constructor(private client: RaiFlowClient) {}
  async publish(block: string, watchConfirmation?: boolean): Promise<{ hash: string }> {
    return this.client.request<{ hash: string }>('POST', '/blocks', {
      block,
      watchConfirmation,
    });
  }
}
```

### 3. `POST /work` — Generate proof-of-work

Generates PoW for a given hash via the RPC pool's work provider.

**Runtime handler addition** (`handler.ts`):
```ts
// POST /work
if (method === 'POST' && parts.length === 1 && parts[0] === 'work') {
  const body = await req.json() as Record<string, unknown>;
  const { hash, difficulty } = body;
  if (typeof hash !== 'string') {
    return errorResponse('Missing required field: hash', 'bad_request', 400);
  }
  const client = runtime.rpcPool?.getClient();
  if (!client) return errorResponse('RPC not configured', 'bad_request', 400);
  const result = await client.workGenerate(hash, typeof difficulty === 'string' ? difficulty : undefined);
  return json(result);
}
```

**SDK addition** — new `WorkResource` class:
```ts
export class WorkResource {
  constructor(private client: RaiFlowClient) {}
  async generate(hash: string, difficulty?: string): Promise<{ work: string }> {
    return this.client.request<{ work: string }>('POST', '/work', { hash, difficulty });
  }
}
```

### Suggested `@openrai/nano-core` Improvement

Currently `@openrai/nano-core` exports `NanoAddress`, `NanoAmount`, `NanoClient`, and `WorkProvider` but not seed derivation. This example uses `nanocurrency` directly for local signing. A future improvement to `@openrai/nano-core` should add:

```ts
export function deriveKeypair(seed: string, index: number): { privateKey: string; publicKey: string };
export function publicKeyToAddress(publicKey: string): string;
export function signBlock(hash: string, privateKey: string): string;
```

This would allow examples to depend solely on `@openrai/nano-core` for crypto primitives. For now, the example depends on `nanocurrency` directly.

# Destination for sweep mode — all funds from every account will be sent here

  - label: "Main Wallet"
    seed: "env:MAIN_WALLET_SEED"

  - label: "Cold Storage"
    seed: "env:COLD_STORAGE_SEED"
    indices: [0]

**Config loader**: A small `loadConfig()` function at the top of `cli.mjs` that:
1. Reads the YAML file
2. Resolves `env:` references (matching RaiFlow convention)
3. Validates required fields
4. Returns a typed config object

Uses the `yaml` npm package (already used by `@openrai/config`). Does NOT import `@openrai/config` to stay lightweight — the example copies the ~10-line `resolveEnv` pattern.

## CLI Modes

| Mode | Invocation | Description |
|------|-----------|-------------|
| Receive (default) | `node cli.mjs` | Check pending, publish receive blocks, print summary table |
| Receive + dry run | `node cli.mjs --dry-run` | Show what would be received without publishing |
| Sweep | `node cli.mjs --sweep` | Receive pending, then send full balance of each account to `sweep_to` |
| Sweep + dry run | `node cli.mjs --sweep --dry-run` | Show what would be swept without publishing |
| Config path | `--config <path>` | Default: `./seed-watcher.yml` |
| Verbose | `--verbose` | Log each API call and block detail |

## Receive Mode Flow

1. Load config, resolve `env:` values, validate
2. For each `(seed, index)` pair in config:
   a. Derive private key and address using `nanocurrency` (`deriveSecretKey`, `derivePublicKey`, `deriveAddress`)
   b. Call `raiflow.accounts.createWatched({ address, label })` — idempotent, registers address for tracking
   c. Call `raiflow.accounts.receivable(accountId)` — get pending blocks for this account
   d. Call `raiflow.accounts.get(accountId)` — get frontier, balance, representative
3. For each pending block found:
   a. Compute new balance: `BigInt(currentBalance) + BigInt(amount)`
   b. Build block using `nanocurrency.createBlock(secretKey, { previous, link: hash, balance, representative, work: null })`
   c. Sign using `nanocurrency.signBlock({ hash: block.hash, secretKey })`
   d. Call `raiflow.work.generate(openFrontier || publicKey)` — get PoW from RaiFlow
   e. Assemble final block JSON with work filled in
   f. Call `raiflow.blocks.publish(blockJson)` — publish to network (skip if `--dry-run`)
4. Re-fetch updated accounts and print summary table

## Sweep Mode Flow

1. Run full receive flow first (steps 1–3 above)
2. Re-fetch all accounts with `raiflow.accounts.get(id)` for updated balances
3. For each account with `balanceRaw > 0`:
   a. Build send block to `sweep_to` address with the entire balance (resulting balance = 0)
   b. Sign with `nanocurrency.signBlock()`
   c. Generate PoW via `raiflow.work.generate(frontier)`
   d. Publish via `raiflow.blocks.publish(blockJson)` (skip if `--dry-run`)
4. Print sweep summary table

## Block Construction Details

All block building and signing happens locally in the CLI using `nanocurrency`. Work generation and publishing go through RaiFlow.

```js
import { deriveSecretKey, derivePublicKey, deriveAddress, createBlock, signBlock } from 'nanocurrency';

// Derive
const secretKey = deriveSecretKey(seed, index);
const publicKey = derivePublicKey(secretKey);
const address = deriveAddress(publicKey);

// For a receive block:
const previous = account.frontier || '0'.repeat(64);  // zeros for open
const newBalance = (BigInt(account.balanceRaw) + BigInt(pendingAmount)).toString(16).padStart(32, '0');
// Note: nanocurrency expects hex balance in createBlock

const block = createBlock(secretKey, {
  previous: previous === '0'.repeat(64) ? '' : previous,
  link: pendingHash,
  balance: newBalance,
  representative: account.representative || config.representative,
  work: null,
});

const signature = signBlock({ hash: block.hash, secretKey });

// Get PoW from RaiFlow
const { work } = await raiflow.work.generate(previous === '0'.repeat(64) ? publicKey : previous);

// Assemble final block
const finalBlock = { ...block.block, work, signature };

// Publish via RaiFlow
await raiflow.blocks.publish(JSON.stringify(finalBlock));
```

**Important**: The `nanocurrency.createBlock` function expects the `previous` field to be an empty string for unopened accounts (not the zero hash). The `balance` field must be the hex representation of the raw balance.

## Error Handling

- Config validation errors → clear messages with field names
- RPC/API errors → per-account error reporting, continue with remaining accounts
- Block publish failures → log the error, mark account, continue
- `--dry-run` → skip all `blocks.publish()` and `work.generate()` calls, log what would happen
- Missing `sweep_to` with `--sweep` → exit with error
- Invalid seed format → exit with error

## Idempotency Notes

- `accounts.createWatched()` is idempotent — returns existing account if already watched
- Receives are naturally idempotent — pending blocks are only listed if not yet received
- Sweep sends use the current frontier as `previous` — a failed send leaves the account unchanged, safe to retry

## Design Principles

- **Never call Nano RPC directly** — all network calls go through RaiFlow SDK
- **Local signing only** — seeds never leave the CLI process; RaiFlow never sees them
- **Exercise RaiFlow capabilities** — watched accounts, receivable discovery, work generation, block publishing
- **Follow project conventions** — ESM, `env:` resolution, workspace deps, no comments unless asked
- **Single-file example** — matches other examples in the repo
