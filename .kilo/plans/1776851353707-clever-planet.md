# RaiFlow Changes: Account + Send Resources with Non-Anemic Orchestration

## Context

Nanosweeper needs two capabilities from RaiFlow:
1. **Account management** — create managed/watched accounts to use as `recipientAccount` when creating invoices
2. **Send orchestration** — queue payouts from a managed account and let RaiFlow handle signing, work generation, block publishing, and confirmation tracking

The current SDK only exposes `InvoicesResource`, `WebhooksResource`, and `SystemResource`. The runtime HTTP handler has no account or send routes. The `AccountStore` and `SendStore` exist in `@openrai/storage` with full SQLite schemas, but they are unused by the runtime prototype.

## Goal

Expose **Account** and **Send** resources through the RaiFlow runtime HTTP API and the `@openrai/raiflow-sdk` package. The Send resource must be **non-anemic**: it does not just store state — it actively signs, publishes, and tracks confirmations via a background orchestrator.

---

## Design Principles

1. **Account resource**: Standard CRUD. Managed accounts derive addresses from the custody seed; watched accounts are external addresses tracked for balance.
2. **Send resource**: The HTTP handler initiates (`queued` state). A background `SendOrchestrator` drives the lifecycle: `queued` → `published` → `confirmed`/`failed`.
3. **Runtime owns the orchestrator**: The orchestrator has dependencies on custodyEngine (signing), rpcPool (RPC + work), sendStore (state), accountStore (balance/address lookup), eventStore (events).
4. **Events are first-class**: Every state transition emits a typed event (`send.queued`, `send.published`, `send.confirmed`, `send.failed`) that flows through the event bus to webhooks.
5. **No anemic DTOs**: The `Send` model carries behavior-relevant fields (accountId, destination, amountRaw, status, blockHash, idempotencyKey). The orchestrator mutates it, not a separate service layer.

**Background orchestration**: Add `SendOrchestrator` class (can live in `packages/runtime/src/send-orchestrator.ts` or inline in `runtime.ts`):

```typescript
class SendOrchestrator {
  constructor(
    private sendStore: SendStore,
    private accountStore: AccountStore,
    private custodyEngine: CustodyEngine,
    private rpcPool: RpcPool,
    private emitEvent: (event: RaiFlowEvent) => Promise<void>,
  ) {}

  private timer: ReturnType<typeof setInterval> | undefined;

  start(intervalMs = 5000): void {
    this.timer = setInterval(() => this.tick(), intervalMs);
  }

  stop(): void {
    if (this.timer) clearInterval(this.timer);
  }

  private async tick(): Promise<void> {
    const queued = await this.sendStore.listByStatus('queued');
    for (const send of queued) {
      await this.publishSend(send);
    }
  }

  private async publishSend(send: Send): Promise<void> {
    try {
      // 1. Fetch account info (frontier, balance, representative)
      const account = await this.accountStore.get(send.accountId);
      if (!account) throw new Error('Account not found');

      const client = this.rpcPool.getClient();
      const info = await client.accountInfo(account.address);

      // 2. Compute new balance (current balance - send amount)
      const newBalanceRaw = (BigInt(info.balance) - BigInt(send.amountRaw)).toString();

      // 3. Sign the block
      const signed = await this.custodyEngine.signSend(
        account.address,
        send.destination,
        newBalanceRaw,     // balance AFTER this send
        info.frontier,
      );

      // 4. Generate work
      const work = await this.custodyEngine.generateWork(signed.hash);

      // 5. Build final block JSON with work included
      const blockJson = JSON.parse(signed.contents);
      blockJson.work = work;

      // 6. Publish to network
      const result = await client.process(JSON.stringify(blockJson));

      // 7. Update send to published
      const published = await this.sendStore.update(send.id, {
        status: 'published',
        blockHash: result.hash,
        publishedAt: new Date().toISOString(),
      });

      // 8. Update account frontier
      await this.accountStore.update(account.id, {
        frontier: result.hash,
        balanceRaw: newBalanceRaw,
      });

      await this.emitEvent({
        id: randomUUID(),
        type: 'send.published',
        timestamp: new Date().toISOString(),
        data: { send: published },
        resourceId: published.id,
        resourceType: 'send',
      });

    } catch (err) {
      // On failure, mark as failed and emit event
      const failed = await this.sendStore.update(send.id, {
        status: 'failed',
      });

      await this.emitEvent({
        id: randomUUID(),
        type: 'send.failed',
        timestamp: new Date().toISOString(),
        data: { send: failed, reason: err instanceof Error ? err.message : String(err) },
        resourceId: failed.id,
        resourceType: 'send',
      });
    }
  }
}
```

**Note on balance tracking**: The `signSend` method in `custodyEngine` takes `balance` as the *post-transaction* balance, which is the Nano protocol convention. The orchestrator computes this by subtracting `amountRaw` from the current account balance.

**Note on frontier**: After successful publish, the account's frontier is updated to the new block hash. This is critical for subsequent sends from the same account.

#### 2b. HTTP Handler (`packages/runtime/src/handler.ts`)

Add route group under `parts[0] === 'accounts'`:

| Method | Path | Action |
|---|---|---|
| POST | `/accounts` | Create account (body has `type: 'managed' \| 'watched'`) |
| GET | `/accounts` | List accounts (optional `?type=managed\|watched`) |
| GET | `/accounts/:id` | Get account by ID |
| PATCH | `/accounts/:id` | Update account label/representative |
| POST | `/accounts/:id/sends` | Queue a send from this managed account |
| GET | `/accounts/:id/sends` | List sends for this account |
| GET | `/sends/:id` | Get send by ID |

**`POST /accounts` request body**:
```json
{
  "type": "managed",
  "label": "nanosweeper-pool",
  "representative": "nano_...",
  "idempotencyKey": "..."
}
```
or
```json
{
  "type": "watched",
  "address": "nano_...",
  "label": "player-wallet"
}
```

**`POST /accounts/:id/sends` request body**:
```json
{
  "destination": "nano_...",
  "amountRaw": "18000000000000000000000000000",
  "idempotencyKey": "game-abc-123-payout"
}
```

Validation rules:
- `type` required for account creation, must be `managed` or `watched`
- `address` required for `watched`, must be valid Nano address
- `destination` required for sends, must be valid Nano address
- `amountRaw` required for sends, must be positive numeric string
- `idempotencyKey` required for sends (prevents double-payout)
- Only `managed` accounts can queue sends; `409 Conflict` if attempted on watched
- Returns `400` for validation failures, `404` for missing accounts, `409` for idempotency hits

- Wire `accountStore` and `sendStore` into `Runtime` config

import type { RaiFlowClient } from '../client.js';

export interface CreateManagedAccountOptions {
  label?: string;
  representative?: string;
  idempotencyKey?: string;
}

export interface CreateWatchedAccountOptions {
  address: string;
  label?: string;
}

export interface ListAccountsOptions {
  type?: 'managed' | 'watched';
}

export class AccountsResource {
  constructor(private client: RaiFlowClient) {}

  async createManaged(options: CreateManagedAccountOptions): Promise<Account> {
    return this.client.request<Account>('POST', '/accounts', {
      type: 'managed',
      ...options,
    });
  }

  async createWatched(options: CreateWatchedAccountOptions): Promise<Account> {
    return this.client.request<Account>('POST', '/accounts', {
      type: 'watched',
      ...options,
    });
  }

  async list(options?: ListAccountsOptions): Promise<{ data: Account[] }> {
    const params = new URLSearchParams();
    if (options?.type) params.set('type', options.type);
    const query = params.toString();
    const path = query ? `/accounts?${query}` : '/accounts';
    return this.client.request<{ data: Account[] }>('GET', path);
  }

  async get(id: string): Promise<Account> {
    return this.client.request<Account>('GET', `/accounts/${id}`);
  }

  async update(id: string, patch: UpdateAccountRequest): Promise<Account> {
    return this.client.request<Account>('PATCH', `/accounts/${id}`, patch);
  }
}

## Why This Is Not Anemic

| Anemic smell | Our approach |
|---|---|
| DTOs with no behavior | `Send` is a state machine. The orchestrator mutates it through real protocol operations (sign → work → publish → confirm). |
| Service layer does everything | `Runtime` coordinates, but `SendOrchestrator` owns the send-specific protocol logic. Account creation uses `custodyEngine` for derivation. |
| HTTP handler just passes through | Handler validates and initiates; the orchestrator drives async lifecycle independently. |
| Store is just CRUD | `SendStore.listByStatus` enables the orchestrator's polling loop. `accountStore` tracks frontier for chaining sends. |
| No side effects | Every state transition emits an event. Webhooks fire automatically. Balance and frontier are updated on publish. |

## Data Flow: Nanosweeper Payout

```
Nanosweeper Worker
  POST /accounts/:id/sends
    { destination: "nano_player", amountRaw: "18000000000000000000000000000", idempotencyKey: "game-123" }
  → Send created with status: 'queued'
  → Event 'send.queued' emitted → webhook to Nanosweeper

SendOrchestrator (background, every 5s)
  1. Fetch account info (frontier, balance) via RPC
  2. Sign send block via custodyEngine
  3. Generate work via custodyEngine
  4. Publish block via RPC process
  5. Update send status: 'published', blockHash set
  6. Update account frontier and balance
  7. Event 'send.published' emitted → webhook to Nanosweeper

Watcher (WebSocket confirmation)
  1. Sees blockHash confirmed on network
  2. Calls Runtime.handleConfirmedBlock()
  3. Runtime looks up send by blockHash
  4. Updates send status: 'confirmed'
  5. Event 'send.confirmed' emitted → webhook to Nanosweeper
```

## Open Decisions

### D1: Watcher integration scope

The `Watcher` from `@openrai/watcher` is not currently instantiated in `main.ts`. Two options:

- **Option A (minimal)**: Worker polls `GET /sends/:id` until status is `confirmed`. No watcher needed for Phase 1.
- **Option B (full)**: Instantiate `Watcher` in `main.ts`, subscribe all managed/watched accounts, use `handleConfirmedBlock` to auto-transition sends and update balances.

**Recommendation**: Option B. Without it, the account balance in the DB becomes stale after sends. The watcher is already a dependency; wiring it up is ~30 lines in `main.ts` plus extending `handleConfirmedBlock`.

### D2: Send confirmation tracking

The `handleConfirmedBlock` method currently only matches incoming payments to invoices. We need to extend it:

1. Check if `block.recipientAccount` matches any account address → update balance
2. Check if `block.blockHash` matches any `send.blockHash` with status `published` → update to `confirmed`

This requires `accountStore.getByAddress()` and `sendStore.getByBlockHash()` (the latter may need adding to the interface).
