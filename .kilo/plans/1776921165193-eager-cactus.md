# Plan: RaiFlow Container Image & Deployment Quickstart

## Overview

Publish the RaiFlow runtime as a container image to `ghcr.io/openrai/raiflow`, add a dead-simple Docker Compose example, and document a "Deployment Quickstart" flow in the README. This also requires implementing API key auto-generation and enforcement (currently the `daemon.apiKey` config field is parsed but never checked), plus a retrievable API key mechanism for Docker deployments.

---

## 1. API Key Auto-Generation & Enforcement

### 1a. New module: `packages/runtime/src/auth.ts`

- If `config.daemon.apiKey` is set (via `RAIFLOW_API_KEY` env or YAML), use it.
- If not set, auto-generate a 256-bit random hex key using `node:crypto`.
- Write the key to `<dirname(config.storage.path)>/.api-key` (derived from the configured SQLite path, so it lands next to the database on the persistent volume).
- On subsequent boots, read the existing key from the file instead of regenerating.
- Log the key location on first generation (not the key itself).
- Export `resolveApiKey(config)` → returns the effective API key string and whether it was newly generated.

### 1b. Auth middleware in `packages/runtime/src/handler.ts`

- `createHandler(runtime, apiKey: string)` — accept the resolved API key as a parameter.
- Before routing, check `Authorization: Bearer <key>` header.
- Exempt `GET /health` from auth (health checks must work without credentials).
- Return `401 { error: { message, code: "unauthorized" } }` on auth failure.
- If `apiKey` is empty/falsy (dev mode with no key configured), skip auth entirely (backward compatible for local dev).

### 1c. CLI command: `packages/runtime/src/show-api-key.ts`

- New entry point that loads config, resolves the key the same way as `main.ts`, and prints it to stdout.
- Added as a second bin in `packages/runtime/package.json`: `"show-api-key": "./dist/show-api-key.js"`.
- Enables: `docker exec <container> show-api-key`

### 1d. Update `main.ts`

- After loading config and opening the database, call `resolveApiKey(config)`.
- Pass the resolved key to `createHandler(runtime, apiKey)`.
- Log whether the key was loaded from config, loaded from file, or auto-generated.

## 2. Dockerfile (project root)

Multi-stage build:

```dockerfile
# Stage 1: Build
FROM node:22-bookworm AS builder
RUN corepack enable && corepack prepare pnpm@10 --activate
WORKDIR /app
# Copy workspace root files first (layer caching)
COPY pnpm-lock.yaml pnpm-workspace.yaml package.json ./
# Copy all package.json files for dependency install
COPY packages/*/package.json ./packages/*/
# Install all dependencies (including dev for build)
RUN pnpm install --frozen-lockfile
# Copy source
COPY packages/ ./packages/
# Build all packages
RUN pnpm -r build
# Remove dev dependencies for production image
RUN pnpm prune --prod

# Stage 2: Runtime
FROM node:22-bookworm-slim
WORKDIR /app
# Copy production node_modules, packages, and root config
COPY --from=builder /app/node_modules/ ./node_modules/
COPY --from=builder /app/packages/ ./packages/
COPY --from=builder /app/package.json /app/pnpm-lock.yaml /app/pnpm-workspace.yaml ./
# Embedded default config for Docker (all secrets via env:)
COPY packages/runtime/docker/raiflow.yml /app/raiflow.yml
# Data volume for SQLite and API key
VOLUME /data
ENV NODE_ENV=production
ENV RAIFLOW_CONFIG_PATH=/app/raiflow.yml
EXPOSE 3100
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD node -e "fetch('http://localhost:3100/health').then(r=>r.ok?process.exit(0):process.exit(1)).catch(()=>process.exit(1))"
CMD ["node", "packages/runtime/dist/main.js"]
```

Node 22 (current active LTS, supported until April 2027) is used consistently across the Dockerfile and CI/release workflows. The `better-sqlite3` native binary is compiled in the builder stage (which has build tools) and the `.node` addon carries through; `bookworm-slim` has the required glibc/libstdc++ runtime libs.

## 4. Docker Default Config: `packages/runtime/docker/raiflow.yml`

```yaml
daemon:
  host: "0.0.0.0"
  port: 3100
  # apiKey: auto-generated if RAIFLOW_API_KEY is unset

nano:
  rpc:
    - "env:NANO_RPC_URL"
  ws: []
  work: []

invoices:
  defaultExpirySeconds: 3600
  autoSweep: false

storage:
  driver: "sqlite"
  path: "/data/raiflow.db"

webhooks: []

logging:
  level: "info"
  format: "json"
```

Note: `NANO_RPC_URL` is the **one required env var**. The config will fail to start if it's not set (per existing `resolveEnv` behavior). `NANO_WS_URL` and `NANO_WORK_URL` are omitted because they default to empty arrays and the runtime works without them. Custody is omitted entirely (also nullable by default).

## 5. Docker Compose Example: `docker-compose.yml` (project root)

```yaml
services:
  raiflow:
    image: ghcr.io/openrai/raiflow:latest
    restart: unless-stopped
    environment:
      NANO_RPC_URL: "https://rpc.nano.org"
      # RAIFLOW_API_KEY:        # optional; auto-generated if unset
      # RAIFLOW_CUSTODY_SEED:   # required only for managed accounts
      # RAIFLOW_CUSTODY_REP:    # required only for managed accounts
    volumes:
      - raiflow-data:/data
    ports:
      - "127.0.0.1:3100:3100"  # Bind to localhost only — do NOT expose to public internet

volumes:
  raiflow-data:
```

Key design decisions:
- **`127.0.0.1:3100:3100`** — RaiFlow is an app-internal service. The port should only be reachable on the host's loopback interface, never forwarded to the public internet.
- **`raiflow-data` volume** — Persists the SQLite database and the auto-generated API key file.
- **Single required env var** — `NANO_RPC_URL`. Everything else is optional.

## 6. CI Workflow: `.github/workflows/container.yml`

```yaml
name: Container

on:
  push:
    branches: [main]
    tags: ['@openrai/runtime@*']
  pull_request:
    branches: [main]

permissions:
  contents: read
  packages: write

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: docker/setup-buildx-action@v3

      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ github.token }}

      - uses: docker/build-push-action@v6
        with:
          context: .
          push: ${{ github.event_name != 'pull_request' }}
          tags: |
            ghcr.io/openrai/raiflow:latest
            ghr.io/openrai/raiflow:${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

- On push to `main`: pushes `latest` + commit SHA tags.
- On PR: builds only (no push) to verify the Dockerfile stays green.
- Uses GitHub Actions cache for Docker layers (`cache-from/to: type=gha`).
- Uses `ghcr.io/openrai/raiflow` as the image name (matching the GitHub repo `OpenRai/RaiFlow`).
- Authentication uses `GITHUB_TOKEN` (automatic, no secrets needed).
- The `packages:write` permission enables pushing to GHCR.

## 7. README.md — New "Deployment Quickstart" Section

Add a section after "Running The Current Code" (or replace it, since Docker is the preferred deployment path). Content:

### Structure

1. **Quick Start** — The `docker compose up` flow with the compose file.
2. **Required Environment Variables** — Table with `NANO_RPC_URL` (required), `RAIFLOW_API_KEY` (optional, auto-generated), `RAIFLOW_CUSTODY_SEED` + `RAIFLOW_CUSTODY_REP` (optional, for managed accounts).
3. **Persistent Data** — Volume mount at `/data`; contains `raiflow.db` (SQLite) and `.api-key` (auto-generated key).
4. **Retrieving Your API Key** — `docker exec <container> show-api-key` or `docker exec <container> cat /data/.api-key`.
5. **Port Binding & Security** — RaiFlow is designed as an **app-private** service. **Do NOT forward port 3100 to the public internet.** Bind to `127.0.0.1` so only local application code can reach it. The API key authenticates internal service requests, not public traffic.
6. **Custom Config** — How to mount your own `raiflow.yml` for advanced settings (webhooks, auto-sweep, multi-node RPC).
